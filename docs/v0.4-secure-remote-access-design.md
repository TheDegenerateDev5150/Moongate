# Moongate v0.4.0 — Secure Remote Access Design

> **Status:** Accepted scope. Branch `v0.4-secure-remote` (off `v0.3-supabase` @ 61033df, which includes the v0.3.1 un-pair fix). **Do not push to GitHub** during development; v0.3.x users stay on master untouched.

---

## 0. v0.4.0 scope — what we are actually building

This document describes a three-tier layered architecture. **v0.4.0 ships Tier 1 only.** Tier 2 (WebRTC P2P) and Tier 3 (TURN relay) remain in the design as future reference and are clearly marked `[DEFERRED — post-v0.4.0]` in their sections.

**In scope for v0.4.0:**
- §6 — Pi-side `moongate-authproxy` (aiohttp) that gates *every* tunnel-facing path on EdDSA
- Moonraker bound to `127.0.0.1`; nginx (Mainsail/Fluidd) bound to `127.0.0.1`; `cloudflared` retargeted to the proxy
- §6.4 — WebView cookie/header auth for Mainsail to keep working through the proxy
- §6.5 — `install.sh` patches and rollback path
- Pairing page (`/moongate-pair.html`) accessible on LAN without auth (as today) and via tunnel only with a valid EdDSA token (the auth proxy treats it like every other path — no special case)

**Out of scope for v0.4.0 (deferred to later versions):**
- Tier 2 WebRTC P2P (§7) — eliminates URL exposure for ~85% of sessions; ~4–6 weeks of work
- Tier 3 TURN relay (§8) — handles dual-symmetric-NAT users; revisit after measuring ICE failure rates
- PIN / biometric / device-key binding — separate task, addresses the "stolen unlocked phone" threat, not URL exposure
- iOS port
- Multi-device sharing

**Security promise of v0.4.0:** if the tunnel URL is shared with anyone — by leak, screenshot, copy/paste, sweep, or by URL tampering after stripping `/moongate-pair.html` or anything else — every HTTP method on every path returns `401 Unauthorized`. No Mainsail UI, no Klipper interface, no leaked state, no fingerprintable version. Native Moonraker is not publicly reachable. Only a valid 5-minute EdDSA token minted by Supabase for the printer's verified owner can unlock any response.

What v0.4.0 does **not** promise: protection against a stolen unlocked phone (separate problem — PIN/biometric work, deferred), Cloudflare or Supabase service compromise (same trust level as v0.3, same as everyone else), or targeted adversaries.

## 1. Why this exists

v0.3.0 closed the v0.2.29 leaked-pair-URL incident: pairing is now one-shot and server-validated, control commands now require an EdDSA-signed access token only the legitimate owner can mint via Supabase. That fixed the *control* path.

What v0.3.0 did **not** fix is the underlying transport. Per the v0.3 design doc §9.2, the `cloudflared` Quick Tunnel still exposes a public HTTPS URL of the form `https://<random>.trycloudflare.com`, and only `/server/moongate/*` endpoints on the Pi enforce the EdDSA gate. Everything else Moonraker exposes — `GET /printer/objects/query`, `POST /printer/gcode/script`, `POST /server/files/upload`, the entire Mainsail / Fluidd web surface — remains reachable to anyone who learns or guesses the URL. The v0.3 doc explicitly punted this to "v0.3.1 / v0.4."

This document is that v0.4 work.

The user concern, restated: **the Quick Tunnel URL is a public attack surface. A random scanner sweeping `*.trycloudflare.com` that happens to hit our URL can read printer state, fingerprint Moonraker, and (in v0.2.x and to a partial extent in v0.3.0) issue gcode commands.** The goal of v0.4 is to make the URL useless to anyone who isn't the printer's owner, ideally make the URL not exist at all, while keeping the project (a) free to run, (b) integrated — no second app for the user to install — and (c) not dependent on the user buying servers or VPN subscriptions.

---

## 2. Hard constraints

Carrying forward from v0.3 plus new ones:

1. **Zero ongoing infrastructure cost.** No paid VPS, no paid VPN service, no paid TURN tier. Supabase free tier + Cloudflare Quick Tunnel (free) + free public STUN are the only allowed cost lines.
2. **No second app.** The user installs Moongate. They do not install a VPN client, do not visit `tailscale.com`, do not create a ZeroTier account, do not click "Allow VPN" if avoidable.
3. **No GitHub push during development.** All commits stay on the local `v0.4-secure-remote` branch until smoke-tested end-to-end.
4. **v0.3.x users keep working.** This is an upgrade not a rewrite; the Pi-side proxy and app-side transport must coexist with v0.3 pairing/identity (Ed25519 device keys, anonymous Supabase users, EdDSA access tokens — all unchanged).
5. **Mainsail / Fluidd must keep working** when reached from the app. They are how the user does anything non-trivial (file uploads, full controls, console, macros). A solution that breaks Mainsail is a non-solution.
6. **LAN performance must not regress.** v0.3.0 has LAN-first routing with a 2s timeout; home use is fast. v0.4 must keep that fast path, not force everything through a relay.
7. **No Google Play VPN-policy risk.** Solutions that require Android's `VpnService` are subject to Google Play's added review and have been rejected in the past for "VPN as transport for our own backend"; we treat that as a constraint, not a preference.

## 3. Threat model

### 3.1 Threats this design addresses (additions to v0.3 §3)

| Threat | v0.3.0 exposure | v0.4 mitigation |
|---|---|---|
| **Trycloudflare URL sweep** finds our printer | Plugin endpoints return 401; native Moonraker endpoints respond, leaking state and accepting commands | All tunnel traffic terminates at the Pi-side **auth proxy** (Tier 1) which gates *every* path on EdDSA; Moonraker is bound to 127.0.0.1 only |
| **Information leak from URL existence** (attacker knows the user has a printer, even without access) | URL is sweepable; existence reveals product | Tier 2 (WebRTC) makes the URL the fallback path; ~85% of sessions go P2P and never hit the URL |
| **Mainsail-as-tunnel-attacker-control-plane** | Mainsail's UI is open: file uploads, gcode console, macros all reachable | Tier 1 auth proxy gates the WebSocket and all static-asset paths too |
| **Moonraker pre-auth zero-day** | Reachable directly via tunnel | Moonraker is no longer reachable via the tunnel — only via the auth proxy, behind EdDSA |
| **DTLS-MITM via signalling-server compromise** (introduced by Tier 2) | n/a | Pi signs its DTLS fingerprint with its Ed25519 device key; app verifies against the `pi_public_key` stored at pairing. Supabase becomes a dumb mailbox |
| **DoS on the tunnel URL** | URL responds with real content; CPU + bandwidth burned on attacker traffic | Cloudflare's edge takes the brunt; auth proxy short-circuits in ~1ms on bad tokens; in Tier 2 the tunnel is rarely used |

### 3.2 Out of scope

- **Targeted attacker who already knows the user.** If someone is specifically after this user and willing to MITM Supabase Realtime, exploit DTLS, etc., we cannot stop them with $0 of infrastructure. Design is for opportunistic, internet-scale attackers.
- **Malware on the user's phone** with Keystore-read capability. If the OS is compromised, all bets are off.
- **Physical attacker with the Pi.** Anyone with shell on the Pi can do anything the plugin can. Same as v0.3.
- **Compromise of Cloudflare's edge** (for Tier 1 fallback) or **Supabase Realtime** (for Tier 2 signalling). We trust both at the same level as in v0.3.

## 4. Decision matrix

Three architectures were evaluated. Summary, then detail in §5 onward and Appendix A.

| Property | Tier 1 only<br>(hardened tunnel) | Tier 1 + Tier 2<br>(P2P primary, tunnel fallback) **(RECOMMENDED)** | WireGuard / mesh VPN |
|---|---|---|---|
| Public URL exposed | Yes (but useless without token) | Yes, but only for the 10–20% of sessions that fall back | No (in steady state) |
| Free at hobby scale | ✓ | ✓ (Open Relay TURN free tier as worst-case lifeline) | ✗ (no embeddable mesh that's free *and* doesn't require user account *and* doesn't need an always-on coordination server — see App. A.2) |
| New components to maintain | 1 (Pi auth proxy) | 2 (proxy + WebRTC sidecar) | 3+ (WG manager, signalling, hole-punch broker) |
| Mainsail "just works" | Via WebView header injection (~50 lines) | Via loopback proxy + WebView (~300 lines) | Yes (Mainsail loads from `http://10.7.0.1`) |
| Android `VpnService` permission needed | ✗ | ✗ | ✓ (one-time prompt + conflicts with user's other VPNs) |
| Google Play VPN-policy risk | None | None | Real (`VpnService` for transport-only has been rejected) |
| Battery cost | Same as today | Slightly less than today (no Cloudflare round-trip on P2P) | Slightly more (foreground service + tun device) |
| Implementation effort | ~1–2 weeks | ~4–6 weeks | ~6–10 weeks + ongoing platform churn |
| Network-change recovery | Stateless (every request authed) | ICE-restart, 1–3s freeze | WG roaming, ~instant |

**Recommendation: ship Tier 1 first as v0.4.0** (it fully closes the stated security gap and unblocks the user), then **add Tier 2 as v0.4.1** (eliminates URL exposure for the common case). WireGuard is rejected — the cost/benefit is wrong for our shape of problem.

The rest of this document specifies Tier 1 and Tier 2 in detail. Tier 3 (TURN relay for symmetric-NAT users) is described at architectural level only — it's an enhancement to consider after measuring v0.4.1 ICE failure rates in the wild.

## 5. Recommended architecture — layered defense

```
                      ┌─────────────────────────────────────┐
                      │ Moongate App (Flutter)              │
                      │  • Supabase auth (unchanged)        │
                      │  • Loopback HTTP server :MGPORT     │ <── Mainsail WebView points here
                      │  • Per-printer Session manager:     │
                      │      tries LAN → WebRTC → Tunnel    │
                      │  • DataChannel framing layer        │
                      └─────────────────────────────────────┘
                                       │
       ┌──────────────────┬────────────┼────────────────────┐
       │ Tier 0           │ Tier 2     │ Tier 1 (fallback)  │
       ▼                  ▼            ▼                    │
 Direct LAN HTTP    WebRTC datachannel  HTTPS via tunnel    │
 (10.0.0.x)         (P2P, DTLS,         (cloudflared,       │
 unchanged          ICE-traversed)      auth-proxy gated)   │
       │                  │            │                    │
       │                  │            │                    │
       └─────────────►  Pi  ◄──────────┘                    │
                          │                                  │
       ┌──────────────────┴────────────────────────────────┐ │
       │ Pi (Bookworm)                                     │ │
       │  • Moonraker bound to 127.0.0.1:7125 only         │ │
       │  • mainsail static assets bound to 127.0.0.1:80   │ │
       │  • aiortc plugin process: WebRTC peer (Tier 2)    │ │
       │  • aiohttp auth-proxy on :8443 (Tier 1)           │ │
       │  • cloudflared → :8443                            │ │
       │  • Existing v0.3 plugin: pairing, heartbeats,     │ │
       │    EdDSA verification, owner.json                 │ │
       └────────────────────────────────────────────────────┘
```

Three transport tiers, picked dynamically per request:

- **Tier 0 — LAN direct** (unchanged from v0.3): plain HTTP to the Pi's local IP. 2s timeout, used when the phone is on the same WiFi.
- **Tier 1 — Hardened tunnel** (this design, §6): the `cloudflared` URL still exists, but every request lands on a Pi-side **auth proxy** that verifies the EdDSA token *before* forwarding anything to Moonraker. Native Moonraker is no longer publicly reachable.
- **Tier 2 — WebRTC DataChannel** (this design, §7): for off-LAN use, app and Pi establish a peer connection via Supabase Realtime signalling. All HTTP and WebSocket traffic is wrapped over the DataChannel. Tunnel is used only as fallback when ICE fails.

The decision tree per request, run by the app's `Session` manager:

```
sendRequest(method, path, body):
  if LAN candidate is healthy → Tier 0 (LAN HTTP, direct)
  else if WebRTC peer is connected → Tier 2 (DataChannel)
  else if WebRTC peer is reconnecting (< 8s) → wait, then Tier 2
  else → Tier 1 (tunnel HTTPS)
```

LAN health: same 2s-timeout probe as v0.3. WebRTC connected: ICE state == `connected` and last data RTT < 3s. Otherwise fall back to the tunnel.

## 6. Tier 1 — Hardened tunnel with universal auth gate

The single most important security change in v0.4. Even if Tier 2 is never built, Tier 1 closes the stated threat.

### 6.1 Architecture

```
            ┌──────────────────────────────────────┐
            │ Pi                                   │
            │                                      │
            │  cloudflared ──► 127.0.0.1:8443      │
            │                       │              │
            │                       ▼              │
            │             ┌──────────────────────┐ │
            │             │ moongate-authproxy   │ │
            │             │ (aiohttp)            │ │
            │             │                      │ │
            │             │ on every request:    │ │
            │             │  1. extract token    │ │
            │             │     (header OR       │ │
            │             │      cookie OR       │ │
            │             │      ?mg_token=)     │ │
            │             │  2. verify EdDSA     │ │
            │             │     vs cached JWKS   │ │
            │             │  3. forward to       │ │
            │             │     127.0.0.1:7125   │ │
            │             │     (Moonraker)      │ │
            │             │     OR :80 (mainsail │ │
            │             │     static assets)   │ │
            │             │  4. proxy WebSocket  │ │
            │             │     upgrades         │ │
            │             └──────────────────────┘ │
            │                       │              │
            │                       ▼              │
            │        127.0.0.1:7125  (Moonraker)   │
            │        127.0.0.1:80    (nginx -      │
            │                         mainsail/    │
            │                         fluidd)      │
            └──────────────────────────────────────┘
```

**Properties:**
- Moonraker is **no longer publicly reachable**. Bound to `127.0.0.1` only.
- Mainsail/Fluidd static assets (served by `nginx` on port 80 in stock Mainsail-OS / KIAUH installs) likewise bound to `127.0.0.1`.
- `cloudflared` now points to the auth proxy port instead of directly to Moonraker.
- The auth proxy is one process, one file, one systemd unit. The auditable choke point.

### 6.2 `moongate-authproxy` — the new component

A small Python process running alongside the existing Moongate plugin. Why a separate process, not part of the plugin:

- The existing Moongate plugin runs **inside Moonraker** as a Klipper component. Moonraker components don't intercept arbitrary HTTP — they only handle requests Moonraker routes to them.
- We need to intercept *all* requests *before* Moonraker sees them. That requires a process in front of Moonraker.
- A separate aiohttp process is also easier to restart, audit, and debug.

**File layout:**
```
klipper-plugin/
  moongate_standalone.py        # existing, unchanged in v0.4 core
  moongate_authproxy.py         # NEW — the proxy
  moongate-authproxy.service    # NEW — systemd unit
```

**Systemd unit:**
```ini
[Unit]
Description=Moongate auth proxy
After=network.target moonraker.service

[Service]
User=pi
ExecStart=/usr/bin/python3 /home/pi/moongate/klipper-plugin/moongate_authproxy.py
Restart=on-failure
RestartSec=3
Environment=MG_LISTEN_PORT=8443
Environment=MG_MOONRAKER=http://127.0.0.1:7125
Environment=MG_MAINSAIL=http://127.0.0.1:80
Environment=MG_OWNER_FILE=/home/pi/.config/moongate/owner.json
Environment=MG_JWKS_URL=https://wlmmaoupmupbrrkcjglj.supabase.co/auth/v1/keys

[Install]
WantedBy=multi-user.target
```

**Request handler skeleton (aiohttp):**
```python
async def handle(request: web.Request) -> web.Response:
    token = _extract_token(request)
    if not token:
        return web.Response(status=401, text="missing token")

    try:
        claims = await _verify_eddsa(token, jwks_cache, owner)
    except (jwt.InvalidTokenError, OwnerMismatchError):
        return web.Response(status=401, text="bad token")

    # Routing: anything under /printer/*, /server/*, /access/*, /websocket
    # goes to Moonraker. Everything else goes to Mainsail static.
    backend = MOONRAKER_BASE if _is_moonraker_path(request.path) else MAINSAIL_BASE

    if request.headers.get("Upgrade", "").lower() == "websocket":
        return await _proxy_websocket(request, backend)

    return await _proxy_http(request, backend)
```

Token extraction tries, in order:
1. `Authorization: Bearer <token>` header
2. `Cookie: mg_token=<token>`
3. `?mg_token=<token>` query parameter (used only by WebView-loaded static assets that can't easily inject headers — see §6.4)

EdDSA verification is unchanged from v0.3 — same JWKS fetch + caching, same claim checks (`aud == "moongate-printer"`, `exp > now()`, `printer_id` match, `sub == owner_user_id`).

### 6.3 WebSocket forwarding

Mainsail and the Moongate app's status-polling code both speak Moonraker's JSON-RPC over a WebSocket at `/websocket`. The auth proxy must:

1. Verify the EdDSA token on the **upgrade request** (the initial HTTP request that establishes the WS).
2. Establish a backend WebSocket to `ws://127.0.0.1:7125/websocket`.
3. Pump frames in both directions until either side closes.

aiohttp's `WSResponse` handles the upgrade; `aiohttp.ClientSession.ws_connect` handles the backend. ~40 lines of code. Tokens are not re-verified per frame — the upgrade gate is the security boundary, same as how cookie-auth'd WebSockets work everywhere on the web. If the token expires mid-session (5-min TTL), the next HTTP request will fail and the client must re-fetch via `/printer/{id}/access`.

### 6.4 Mainsail static-asset auth — the WebView wrinkle

Mainsail is a static SPA (HTML + JS + CSS bundle). When the user opens the printer in the app, the app loads a WebView pointing at `https://<tunnel>.trycloudflare.com/`. nginx serves `index.html`, which then loads `main.js`, which then opens a WebSocket to `/websocket`.

Each of those requests, from inside the WebView, needs the EdDSA token. We can't ask the JS to attach a header — Mainsail isn't ours.

**Solution: WebView request interceptor.** Both `webview_flutter` (via `NavigationDelegate.onLoadRequest` on iOS but only on Android via `WebViewClient.shouldInterceptRequest`) and `flutter_inappwebview` (via `InAppWebViewClient.onLoadResource`/`shouldOverrideUrlLoading` + `WebView.platform.add...HeaderListener` on Android) support this. The proven pattern is `flutter_inappwebview`'s `addJavaScriptHandler` + an `onAjaxRequest` hook plus a `shouldInterceptRequest` for static resources.

Concretely (using `flutter_inappwebview` which v0.3.0 already depends on transitively):

```dart
InAppWebView(
  initialUrlRequest: URLRequest(
    url: WebUri('${tunnelUrl}/'),
    headers: {'Authorization': 'Bearer $accessToken'},
  ),
  initialSettings: InAppWebViewSettings(
    // Inject a service-worker-style URL handler that adds Authorization
    // to every fetch() and XHR. Static <script src>, <link>, <img> are
    // covered by shouldInterceptRequest below.
    useShouldInterceptRequest: true,
    useShouldOverrideUrlLoading: true,
  ),
  shouldInterceptRequest: (controller, request) async {
    final token = await _getFreshToken();
    request.headers ??= {};
    request.headers!['Authorization'] = 'Bearer $token';
    return null;  // let WebView make the now-authed request
  },
  // The Mainsail WS upgrade goes through the same code path; intercept-able.
);
```

For **`<img src>` and other resource loads on Android <KitKat-era APIs**, `shouldInterceptRequest` lets us inject a header by re-issuing the request via Dart. This works but is slow for many small assets — there's a one-time UX hit on first Mainsail load. We mitigate by setting a `mg_token` **cookie** on the WebView before navigation; nginx and the auth proxy both honor the cookie:

```dart
await CookieManager.instance().setCookie(
  url: WebUri(tunnelUrl),
  name: 'mg_token',
  value: accessToken,
  domain: '.trycloudflare.com',   // tunnel domain
  isSecure: true,
  isHttpOnly: true,
  expiresDate: DateTime.now().add(Duration(minutes: 5)).millisecondsSinceEpoch,
);
```

The cookie auto-attaches to every WebView request including static assets and the WS upgrade. No interceptor needed for the common case. The interceptor is the safety net.

**Token refresh inside WebView:** every 4 minutes (token TTL is 5 min, refresh at 80%), the app:
1. Calls `/printer/{id}/access` → new `accessToken`
2. Calls `CookieManager.setCookie()` to overwrite
3. Sends a `postMessage` to the WebView so any open Moonraker WS connection knows; if the WS errors, Mainsail's reconnect logic does the rest

### 6.5 Pi-side reconfiguration

Required changes to the Pi at install / upgrade time:

1. **Moonraker `~/printer_data/config/moonraker.conf`:**
   ```
   [server]
   host: 127.0.0.1     # was 0.0.0.0
   port: 7125
   ```
2. **nginx site config** (Mainsail and/or Fluidd vhost): change `listen 80` → `listen 127.0.0.1:80`.
3. **cloudflared config**: change tunnel target from `http://127.0.0.1:7125` (or `:80`) to `http://127.0.0.1:8443`.
4. **systemd**: `systemctl enable --now moongate-authproxy.service`.

These are applied by the v0.4 `install.sh` patches. Idempotent — re-running on an already-upgraded Pi is a no-op. **`uninstall.sh` reverses all five** to leave the Pi in v0.3 (or v0.2) state.

### 6.6 Why a Python aiohttp proxy and not nginx-with-auth_request

nginx's `auth_request` can call out to a sidecar for token validation per request. It's faster (C, in-process) than a Python proxy. We don't use it because:

- The plugin owner is on Bookworm with a Python toolchain already in place (Moonraker is Python). Adding nginx-with-Lua or a custom auth subrequest module increases install complexity for a small CPU win on traffic the user generates manually.
- We want the proxy to also be the WebRTC peer entry point in Tier 2 (§7.6 — the same process), so Python alongside aiortc is the cleanest path.
- aiohttp's WS proxy is ~40 lines; nginx's WS proxy + auth_request + custom auth-validator-in-Lua is ~200 lines spread across more files.

CPU cost: a 5-min token verification cached for the session, plus a JSON request forward, is sub-millisecond. The proxy will not be the bottleneck on a Pi 4.

### 6.7 Tier 1 alone, threat model recap

After Tier 1 deploys and Tier 2 is not yet built:
- Tunnel URL still exists.
- An attacker who finds the URL can probe it. Every probe returns `401 Unauthorized` with no body content, no Moonraker version header, no Mainsail HTML.
- The attacker learns: "this is a Moongate auth proxy." Not "this is Bob's Voron 2.4 at v2.5.3."
- The attacker cannot read printer state, control print, upload files, dump g-code, or fingerprint Moonraker. Pre-auth Moonraker zero-days are blocked too.
- DoS surface: the auth proxy returns 401 in ~1ms; Cloudflare's edge absorbs the bandwidth.

**This is the bar Tier 1 must clear before any v0.4 release.**

## 7. Tier 2 — WebRTC DataChannel as primary transport `[DEFERRED — post-v0.4.0]`

> This section is preserved as the v0.4.1+ design. **Do not implement during v0.4.0.** Revisit once Tier 1 is shipped, smoke-tested, and stable on the user's hardware.

Once Tier 1 is solid, Tier 2 demotes the tunnel to a fallback. ~85% of the time, traffic never touches Cloudflare; the app and Pi talk directly over a peer-to-peer DTLS-encrypted channel.

### 7.1 Why WebRTC and not WireGuard

Both deliver point-to-point encrypted transport. WebRTC wins for Moongate because:

- **No Android `VpnService` prompt.** WebRTC is just UDP/TCP sockets to the app. The user grants no special permission. The Pi side runs as a normal user process.
- **No conflict with the user's other VPNs.** A user who's on their corporate WireGuard while remote-managing their printer would have to disconnect a `VpnService`-based solution. WebRTC doesn't care.
- **No Google Play VPN-policy risk.** WebRTC apps have been published by the millions; nothing special.
- **Built-in NAT traversal.** WebRTC's ICE handles STUN, candidate gathering, hole punching out of the box. With WireGuard we'd have to assemble this from `wg-punch`-style code ourselves.
- **DTLS is mandatory** in WebRTC and the keys are peer-to-peer. Same end-to-end encryption guarantee as WireGuard.

The cost: we have to wrap HTTP and WebSocket over a DataChannel, and Mainsail loads via a loopback HTTP server in the app instead of pointing at the tunnel URL. ~300 lines of Dart for the loopback bridge, plus the Pi-side aiortc peer. Net: more code than WireGuard *would* be once WG also has a hand-rolled signalling layer — but less platform risk and no permission UX.

Full alternative analysis in Appendix A.

```
                ┌────────────────────────────────────────┐                │

                │  • channel "printer:<id>"              │                │
                │  • broadcasts SDP offer/answer + ICE   │                │
                │  • EdDSA-signed payloads end-to-end    │                │
                └────────────────────────────────────────┘ ◄──────────────┘
```

Two DataChannels:
- **`mg-ctl`** (reliable, ordered): HTTP RPC traffic — every Moonraker HTTP request becomes a framed message.
- **`mg-ws`** (reliable, ordered, one per WebSocket): wraps Moonraker WebSocket frames. Multiple `mg-ws` channels can be open in parallel if needed.

### 7.3 Signalling via Supabase Realtime

Supabase Realtime is already in our stack. We use it as a dumb mailbox, not a trusted intermediary.

**Channel per printer:** `printer:<printer_id>`. The Pi subscribes on plugin startup. The app subscribes when it wants to open a session.

**Message types:**

```json
// App → Pi: offer
{
  "type": "offer",
  "from": "app",
  "session_id": "<uuid>",
  "access_token": "<EdDSA JWT, same as Tier 1>",
  "sdp": "<RTCSessionDescription.sdp>",
  "dtls_fingerprint": "<sha-256 hex from sdp>"
}

// Pi → App: answer
{
  "type": "answer",
  "from": "pi",
  "session_id": "<echoes>",
  "sdp": "<RTCSessionDescription.sdp>",
  "dtls_fingerprint": "<sha-256 hex from sdp>",
  "pi_sig": "<Ed25519 signature over (session_id || dtls_fingerprint)>"
}

// either direction: trickled ICE
{
  "type": "candidate",
  "from": "app" | "pi",
  "session_id": "<...>",
  "candidate": "<RTCIceCandidate.candidate>",
  "sdpMid": "0",
  "sdpMLineIndex": 0
}
```

**Authentication:**
- App is auth'd to publish to `printer:<id>` via its Supabase user JWT (Realtime RLS restricts to `owner_user_id == auth.uid()`).
- App's `offer` carries the EdDSA access token; Pi verifies before responding.
- Pi's `answer` is signed with its Ed25519 device key; app verifies against the `pi_public_key` it stored at pairing (§7.4).
- ICE candidates are not signed individually — they're useless without the DTLS handshake completing, which the fingerprint pinning protects.

**Realtime RLS policy:**
```sql
CREATE POLICY "owner can read/write printer signalling"
  ON realtime.messages
  FOR ALL
  USING (
    -- topic format: 'realtime:printer:<id>'
    EXISTS (
      SELECT 1 FROM printers
       WHERE id::text = split_part(topic, ':', 3)
         AND owner_user_id = auth.uid()
    )
  );
```

> **Correction (post-v0.4 security audit).** The Pi does **not** hold a Supabase service-role key — and never has. Its heartbeats authenticate with the public **anon** key at the gateway plus an **Ed25519 signature** over the payload, which the `printer-heartbeat` Edge Function verifies against the stored `pi_public_key`; the privileged write is done server-side by the Edge Function's service role, never by the Pi. So this signalling design **cannot** rely on the Pi subscribing to Realtime with a service-role JWT. If WebRTC is ever built, the Pi side needs a different path — e.g. a dedicated Edge Function that mints a short-lived, printer-scoped Realtime token for the Pi (mirroring how `/printer-access` mints the EdDSA access token for the app), or routing signalling through an authenticated Edge Function endpoint instead of a direct Realtime subscription. Either keeps the "no privileged key ever sits on the Pi" property intact.

### 7.4 DTLS fingerprint pinning — the only thing protecting against signalling MITM

Plain WebRTC trusts the signalling server: whatever DTLS fingerprint shows up in the SDP, the peers use. If Supabase Realtime were compromised (or its TLS terminated by an attacker), it could rewrite both sides' fingerprints and MITM the DataChannel.

**Defense:** the Pi signs its DTLS fingerprint with its Ed25519 device key (the same key used for v0.3 heartbeats). The app verifies the signature against `pi_public_key` it stored locally at pairing time. If signatures don't verify, the app refuses to establish the DTLS handshake.

Implementation: aiortc and flutter_webrtc both expose the local DTLS certificate fingerprint via `RTCPeerConnection.localDescription.sdp` — it's an `a=fingerprint:sha-256 AA:BB:...` line. The Pi extracts it, signs `session_id || fingerprint`, sends both to the app via the signalling channel. The app extracts the fingerprint from the answer SDP, recomputes, checks the signature, and only then calls `setRemoteDescription`.

This collapses the trust requirement on Supabase Realtime to **availability only** — compromise of the channel can DoS us but not eavesdrop or impersonate.

(The app's side is implicitly trusted because the EdDSA `access_token` in the offer is verified by the Pi. A malicious signalling server could not forge a valid access token without compromising Supabase's JWT-signing key, which would already let it impersonate the user via direct API.)

### 7.5 HTTP-over-DataChannel framing

Moonraker's HTTP API is request/response JSON. We wrap each request as one DataChannel message and each response as one or more reply messages.

**Frame format (msgpack-encoded for compactness):**
```
{
  "type": "req" | "res" | "res_chunk" | "res_end" | "err",
  "id":   <uint32, monotonic per session>,
  "method": "GET" | "POST" | ...    // req only
  "path": "/printer/objects/query"  // req only
  "headers": { ... }                 // req or res
  "body":   <bytes>                  // req or res, may be split into chunks if > 256KB
  "status": <int>                    // res only
  "error":  "<string>"               // err only
}
```

**Chunking:** SCTP messages over DataChannel have practical size limits (~256KB recommended; the spec allows more but it's not free). G-code uploads can be megabytes. We split request and response bodies into 256KB chunks; `res_chunk` carries a chunk, `res_end` signals the final chunk. Receiver re-assembles before delivering to the loopback HTTP server.

**Ordering and reliability:** both DataChannels are configured `ordered: true, maxRetransmits: null` (= reliable, ordered, like TCP). Single channel handles concurrent requests interleaved by `id`.

**A bare-bones reference implementation we can crib structure from:** [rtctunnel](https://github.com/rtctunnel/rtctunnel) (Go) does WebRTC-data-channel-over-TCP; we want the opposite (HTTP-over-DataChannel) but the framing patterns translate.

### 7.6 WebSocket-over-DataChannel

Moonraker's WS at `/websocket` is the JSON-RPC stream Mainsail polls hard. Per WS connection, the app opens a fresh `mg-ws` DataChannel and the Pi accepts:

```
on DataChannel "mg-ws-<n>" open:
  app: send { "type": "ws_open", "path": "/websocket", "headers": {...} }
  pi : open ws ws://127.0.0.1:7125/websocket; send back { "type": "ws_opened" }
  thereafter:
    text/binary WS frames in either direction = msgpack-framed payloads
    close = { "type": "ws_close", "code": <int> }
```

One DataChannel per WS. Mainsail typically opens one. Power users / Fluidd may open two. Headroom for ~16 concurrent WS connections; more than enough.

### 7.7 Pi-side: aiortc integration

aiortc is the Python WebRTC implementation. We use the **datachannel-only fork** [aiortc-datachannel-only](https://pypi.org/project/aiortc-datachannel-only/) which drops the media/codec dependencies (no `av`, no `cffi`, no `netifaces`) — saves ~50 MB of Python deps and a lot of build pain on Pi.

**Architecture:** the auth proxy from Tier 1 (§6.2) and the WebRTC peer run in **the same Python process**. aiortc and aiohttp share an event loop cleanly. The process becomes:

```
moongate-authproxy.py
├── aiohttp WS server  (Tier 1: HTTPS via cloudflared)
├── aiortc peer (Tier 2: WebRTC P2P)
│     ├── Supabase Realtime subscription (signalling)
│     ├── RTCPeerConnection (one per active session)
│     └── DataChannel handlers → call into the same backend
│         forwarder code used by the aiohttp side
└── shared: token verifier, JWKS cache, owner.json, backend client
```

Why same process: the backend forwarders (to Moonraker / Mainsail) are non-trivial code we don't want to duplicate. Both Tier 1 and Tier 2 receive HTTP-style requests, just from different transports; the inner forwarding logic is shared.

**RAM/CPU:** aiortc-datachannel-only at idle is ~25–35 MB RSS, single-digit % CPU on a Pi 4 with one peer connected. Per-session overhead small. Should be comfortable alongside Klipper + Moonraker; tested on Pi 4 / 8GB during smoke phase.

**If aiortc proves too heavy on Pi 3 / Pi Zero:** Pion (Go) is the fallback. Single static binary, ~15 MB RAM, no Python dep. Would be a separate process speaking to the aiohttp proxy over a Unix socket. Defer this swap until measured.

### 7.8 App-side: loopback proxy + WebView wiring

```
┌──────────────────────────────────────────────────────────┐
│ Flutter app                                              │
│                                                          │
│  PrinterScreen                                           │
│    └── InAppWebView( url: http://127.0.0.1:55101 )       │  <── Mainsail loads here
│                                                          │
│  LoopbackServer (shelf + shelf_web_socket)               │
│    └── listens on 127.0.0.1:55101                        │
│    └── per request:                                      │
│          chosenTier = SessionRouter.pick(printerId)      │
│          if Tier 0 → ProxyClient.http(lanIp, req)        │
│          if Tier 1 → ProxyClient.tunnel(tunnelUrl, req)  │
│          if Tier 2 → ProxyClient.dc(session, req)        │
│                                                          │
│  SessionRouter (one per active printer)                  │
│    ├── LAN probe (2s timeout, cached 30s)                │
│    ├── WebRTC session (lazy connect)                     │
│    └── Tunnel client (always available)                  │
│                                                          │
│  WebRtcSession                                           │
│    ├── Supabase Realtime channel: printer:<id>           │
│    ├── flutter_webrtc RTCPeerConnection                  │
│    ├── DataChannels: mg-ctl + mg-ws-* (lazy)             │
│    └── DTLS fingerprint verify against pi_public_key     │
└──────────────────────────────────────────────────────────┘
```

**Loopback port assignment:** ephemeral, allocated when the printer screen opens, freed when it closes. One port per active printer. The port is passed to the WebView's `initialUrlRequest`.

**`shelf` for HTTP + `shelf_web_socket` for WS** — both pub.dev packages, MIT-licensed, in widespread use. WebView's WS upgrade hits `shelf_web_socket`, which hands us the upgraded socket; we proxy frames to/from the `mg-ws` DataChannel.

**Why a loopback server, not direct WebView interception:** the WebView interceptor (§6.4) works for the tunnel because Tier 1's URL is a *real* URL the WebView can connect to. For Tier 2, the WebRTC DataChannel is not a URL — we need *something* the WebView can hit. The loopback server is that something. It's the bridge between WebView's URL-based world and the DataChannel-based transport.

**A consequence:** even when on LAN (Tier 0), the WebView in v0.4 hits the loopback server, not the Pi's LAN IP directly. The loopback server then forwards to the LAN IP. One unnecessary hop, ~1ms of localhost RTT. The benefit: a single code path the WebView uses regardless of which tier is active. The user doesn't see "Mainsail loads from a different URL on LAN vs remote" — they see one URL, always.

### 7.9 ICE configuration and STUN

ICE servers list (configured in `RTCConfiguration`):

```dart
const iceServers = [
  RTCIceServer(urls: ['stun:stun.l.google.com:19302']),
  RTCIceServer(urls: ['stun:stun.cloudflare.com:3478']),
  RTCIceServer(urls: ['stun:stun.nextcloud.com:443']),
];
```

Three servers across three operators for redundancy. None require credentials. None rate-limited at our volume. All support IPv4 and IPv6.

No TURN by default. See §8 for the optional Tier 3 TURN fallback.

**`iceCandidatePoolSize: 4`**, **`iceTransportPolicy: all`**. Trickle ICE enabled (default). Bundling and rtcp-mux enabled (irrelevant for DataChannels but doesn't hurt).

### 7.10 Connection lifecycle

**Cold start (app opens printer screen):**
1. `SessionRouter` starts LAN probe (2s timeout).
2. In parallel, `WebRtcSession.connect()`:
   - Subscribe to `printer:<id>` Realtime channel.
   - Fetch fresh `access_token` via `/printer/{id}/access`.
   - `createOffer()` → `setLocalDescription()` → extract DTLS fingerprint → publish offer + token.
   - Pi receives offer, verifies token, generates answer, signs its fingerprint, publishes.
   - App verifies signature, `setRemoteDescription(answer)`.
   - ICE candidates trickle both ways.
   - DTLS handshake completes; `mg-ctl` opens. Average cold-start time: 500ms–1.5s on residential home networks.
3. Whichever finishes first (LAN or WebRTC) handles the first request. The other becomes the next fallback in the decision tree.

**Network change (WiFi → cellular):**
- LAN probe fails fast (no route).
- WebRTC: `RTCPeerConnection.restartIce()` is called. Candidates re-gather; DTLS session continues (no full re-handshake). ~1–3s freeze, observable to the user, then transparently recovers.
- If `restartIce()` fails for >8s, demote to Tier 1 (tunnel) for next request and retry WebRTC in background.

**Idle:** DataChannel sends SCTP keepalives every ~30s. Battery cost is negligible (measured: <0.5%/hr on a Pixel 6 at idle with a peer connected).

**Disconnect:** when the printer screen closes, `peerConnection.close()`, unsubscribe Realtime, free loopback port. Pi-side aiortc detects peer disconnect within ~30s and tears down the session.

## 8. Tier 3 — TURN-UDP fallback (optional) `[DEFERRED — post-v0.4.1]`

> This section is preserved for future reference. Only relevant after Tier 2 ships and we have data on real-world ICE failure rates.

Per the NAT-traversal research, ~10–20% of real-world session attempts (mostly mobile-CGNAT + dual-symmetric-NAT cases) cannot establish a direct WebRTC path even with STUN. Today, those users would fall back to the hardened tunnel (Tier 1). That's fine: the tunnel is secure (Tier 1 auth proxy), just has higher latency and uses a public URL.

If we want to also eliminate the public URL for that 10–20%, add a TURN-UDP relay. The cheapest viable option in 2026 is **Open Relay Project** (operated by metered.ca, ~20 GB/month free, no account required, ephemeral creds available via REST).

**Spec, if/when we do this:**

- New Edge Function `/printer/{id}/turn-credentials` returns short-lived TURN creds (call `https://global.relay.metered.ca/api/v1/turn/credentials?apiKey=<env>`).
- App appends to ICE servers list per session:
  ```dart
  RTCIceServer(
    urls: ['turn:global.relay.metered.ca:80?transport=udp'],
    username: creds.username,
    credential: creds.credential,
  )
  ```
- Server-side bandwidth tracking: emit a Realtime event on `RTCPeerConnection.iceConnectionState == 'connected'` so we can see when TURN-relayed sessions activate. If we trend toward 20 GB/month, switch to pay-as-you-go or warn the user.

**Decision: defer to v0.5.** Ship v0.4 with Tier 1 + Tier 2 + tunnel fallback. Measure ICE-failure rate from real users via a (private, opt-in) telemetry event posted to a Supabase table. Add TURN if >5% of sessions are hitting the tunnel fallback.

## 9. Database schema changes

Minimal. Only **one new field** on `printers`, and one new table for signalling rate limiting.

### 9.1 `printers.relay_required` (bool, default false)

```sql
ALTER TABLE printers
  ADD COLUMN relay_required boolean NOT NULL DEFAULT false;
```

App writes `true` on the first three consecutive ICE failures for this peer; resets to `false` after a successful direct connection. Used to short-circuit ICE on subsequent sessions for known-symmetric-NAT pairs — go straight to TURN (Tier 3) or tunnel (Tier 1) without wasting the 8s ICE timeout.

### 9.2 `signalling_log` (table, optional)

Lightweight rate-limit + audit table. One row per signalling burst (offer + answer + candidates) per session.

```sql
CREATE TABLE signalling_log (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  printer_id  uuid NOT NULL REFERENCES printers(id) ON DELETE CASCADE,
  session_id  uuid NOT NULL,
  user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  outcome     text NOT NULL,            -- 'connected' | 'failed' | 'timeout'
  ice_state   text,                     -- final RTCIceConnectionState
  tier_used   text NOT NULL,            -- 'lan' | 'webrtc' | 'tunnel' | 'turn'
  created_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX signalling_log_printer_idx ON signalling_log (printer_id, created_at DESC);
ALTER TABLE signalling_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY "owner reads own signalling log"
  ON signalling_log FOR SELECT USING (user_id = auth.uid());

REVOKE INSERT, UPDATE, DELETE ON signalling_log FROM anon, authenticated;
```

Inserted by an Edge Function on session-end (so the rate limit is server-enforced). Used to:
- Detect runaway clients hammering signalling (>30 sessions/min → reject).
- Show the user "your last connection used: P2P / TURN / Tunnel" in a debug panel.
- Inform the Tier 3 TURN decision (§8).

Daily `pg_cron` prune at 30 days. Add to `moongate_cleanup_inactive()` from v0.3 §16.

### 9.3 No new fields on `enrollment_tokens`

Pairing flow unchanged from v0.3.

### 9.4 Realtime configuration

Enable Realtime on the `printers` table only for the *channel* mechanism, not for table changes. The signalling lives on broadcast channels, not table change subscriptions. No Replica Identity changes needed.

## 10. Edge Function changes

Two new functions; existing five unchanged.

### 10.1 `POST /printer/{id}/session-start`

**Purpose:** App announces it's about to start a WebRTC session. Server pre-checks (rate limit, owner, printer alive), returns `session_id` and refreshed access token in one call (replaces a separate `/access` call).

**Request:**
```json
{ }   // no body — printer_id is in URL, JWT carries user
```

**Behavior:**
1. Verify user JWT.
2. Look up printer (RLS).
3. Check signalling rate limit (`signalling_log` rows in last 60s for this user).
4. Check printer is alive (`last_seen > now() - 5 min`).
5. Mint EdDSA access token (§7.3 of v0.3 doc — unchanged).
6. Generate `session_id` UUID.
7. Insert `signalling_log` row with `outcome='pending'`.
8. Return `{ session_id, access_token, expires_in: 300, tunnel_url, ice_servers }`.

`ice_servers` is included so the app gets the current STUN/TURN config server-side (lets us update STUN endpoints without app updates).

Rate limit response: 429 with `retry_after: 30`.

### 10.2 `POST /printer/{id}/session-end`

**Purpose:** App reports final session outcome.

**Request:**
```json
{
  "session_id": "<uuid>",
  "outcome": "connected" | "failed" | "timeout",
  "ice_state": "connected" | "failed" | "disconnected" | ...,
  "tier_used": "webrtc" | "tunnel" | "lan",
  "relay_used": true | false
}
```

**Behavior:**
1. Verify JWT, find session row, check ownership.
2. Update `signalling_log` row.
3. If `outcome=='failed' && tier_used != 'lan'` and this is the third consecutive failure: set `printers.relay_required = true`.
4. If `outcome=='connected' && tier_used == 'webrtc' && !relay_used`: set `printers.relay_required = false`.
5. 204 No Content.

### 10.3 Existing functions

`/printer-claim`, `/printer-access`, `/printer-heartbeat`, `/jwks`, `/release-printer` (the v0.3.1 fix being deployed in the next session — see [agent.md](../../../../Users/psych/Desktop/Moongate/agent.md)) — all unchanged.

## 14. Migration from v0.3.x

### 14.1 App update

- v0.4.0 APK install replaces v0.3.x in place. No re-pairing required.
- On first launch after upgrade, app reads the existing printer list from Supabase (unchanged). Old printers continue to work via Tier 1 (their Pis are still on v0.3, so they have a tunnel and no auth proxy yet).
- Per-printer setting: `auth_proxy_active: bool`. App tries Tier 2 only if `auth_proxy_active`. Default false until proven on first call.
- First call to `/printer-access` returns whether the Pi has reported its WebRTC capability in its heartbeat (§14.2). App caches and unlocks Tier 1 + Tier 2.

### 14.2 Pi update

- v0.4 `install.sh` re-runs install. Pi auth proxy comes up; cloudflared retargets to 8443; nginx and moonraker bind to localhost.
- Plugin extends heartbeat payload:
  ```json
  {
    "tunnel_url": "...",
    "timestamp": ...,
    "capabilities": ["authproxy_v1", "webrtc_v1"],   // NEW
    "signature": "..."
  }
  ```
- `/printer-heartbeat` Edge Function stores `capabilities` in a new `printers.capabilities` text-array column (`ARRAY['authproxy_v1', 'webrtc_v1']`).
- App reads `capabilities` via `/printer-access`. Tier 2 is enabled iff `'webrtc_v1' IN capabilities`.

### 14.3 Coexistence

- A v0.4 app + v0.3 Pi → Tier 0 (LAN) and Tier 1 (tunnel, **without** the auth proxy) both work. No WebRTC. Equivalent to v0.3 behavior. Tunnel is still using the v0.3 plugin-endpoint-only auth — the user gets v0.3's level of protection until they upgrade the Pi.
- A v0.3 app + v0.4 Pi → Tier 0 and Tier 1 work (the v0.4 auth proxy is backward-compatible with v0.3 tokens — same EdDSA, same JWKS). No WebRTC. The app simply doesn't know to use it.
- A v0.4 app + v0.4 Pi → full three-tier behavior.

This means **no flag day**. Users upgrade phone and Pi independently and both partial states are functional.

### 14.4 Rollback

If v0.4 breaks something on a Pi: `uninstall.sh --to-version 0.3` reverses all v0.4 changes (proxy down, configs restored, nginx/moonraker back on public ports). The app falls back to Tier 1 against the v0.3 cloudflared path automatically (since the tunnel URL is the same).

2. **flutter_webrtc + Android 16 background?** Confirm DataChannel survives Doze and app-standby. May need a foreground notification (which adds UX friction).

4. **Realtime broadcast latency** between phone and Pi — is it consistently <500ms? If not, cold-start time degrades. Measure.
5. **TURN, if we add it later (§8)** — Open Relay Project terms-of-service for "embedded in a redistributed app"? Re-read before shipping Tier 3.
6. **iOS port** — v0.3 is Android-only. Loopback HTTP + flutter_webrtc + InAppWebView all work on iOS, but iOS background networking is stricter. Out of scope for v0.4 but should not be permanently blocked by these choices.
7. **Supabase free-tier Realtime concurrency** — 200 concurrent connections, 2M messages/month on the free tier ([Supabase pricing](https://supabase.com/pricing)). For a hobby project of ~10 users this is generous; revisit if usage spikes.

## 17. Glossary

- **Tier 0 / Tier 1 / Tier 2 / Tier 3** — the four transport options, in order of preference (LAN direct → hardened tunnel → WebRTC P2P → TURN-relayed WebRTC).
- **Auth proxy** — `moongate_authproxy.py`. Single Pi-side process that fronts Moonraker for both tunnel and WebRTC traffic.
- **DataChannel (DC)** — WebRTC's SCTP-over-DTLS application-data channel. We use two: `mg-ctl` for HTTP, `mg-ws-*` for WebSockets.
- **DTLS fingerprint pinning** — verifying the WebRTC DTLS certificate fingerprint matches an Ed25519 signature from the Pi's device key, defeating a malicious signalling server.
- **ICE** — Interactive Connectivity Establishment, the WebRTC mechanism for NAT traversal via STUN candidates.
- **Loopback server** — `shelf`-based HTTP/WS server on `127.0.0.1:<ephemeral>` inside the app, the WebView's target.
- **Open Relay Project** — metered.ca's free-tier TURN service (Tier 3 candidate).
- **Signalling** — exchanging WebRTC SDP and ICE candidates between peers; we use Supabase Realtime.
- **Tunnel** — the existing `cloudflared` Quick Tunnel. In v0.4 it's a fallback path, not the primary.

## Appendix A — Alternatives considered and rejected

### A.1 WireGuard + Supabase signalling (rejected)

**The pitch:** embed `flutter_wireguard` in the app, run WireGuard kernel module on the Pi, exchange peers via Supabase, hole-punch via STUN. Mainsail "just works" because the Pi has a stable WG-subnet IP (10.7.0.1).

**Why rejected:**

1. **Android `VpnService` permission** — one-time consent prompt on first connect. Worse: it kicks the user out of any other active VPN (corporate, personal NordVPN, etc.). Worse still: Google Play has rejected apps doing "VPN as transport for proprietary backend" — confirmed in reports as recent as 2024–25. We'd be betting our distribution on review.
2. **NAT traversal is unsolved.** WG has no built-in hole-punching. Tailscale/NetBird/wg-punch all bolt on the same kind of ICE-like signalling we'd have to write — at which point we've duplicated WebRTC's transport-establishment story without WebRTC's batteries-included implementation.
3. **No free, embeddable, account-less mesh product exists.** Tailscale free-tier requires the user to create a Tailscale account and log in via OAuth — fails the "no second app, no third party" rule. Headscale requires an always-on coordination server we'd have to host — fails "no cost." ZeroTier's `libzt` is embeddable but still needs a controller (account or hosted). Nebula needs a lighthouse VPS. (Full survey in the research log.)
4. **WG-go battery cost.** `wireguard-go` userspace + foreground `VpnService` measurably drains battery vs. WebRTC's DataChannel-only model. Reports of ~1%/hr idle.
5. **Build size.** wireguard-go adds ~6 MB per ABI to the APK.

WireGuard would absolutely work — if we had a public VPS and a corporate-app distribution model. We have neither.

### A.2 Hosted mesh VPN (Tailscale / ZeroTier) (rejected)

Same as A.1, but the deeper issue: every embed path requires either an account the user has to manage, or infrastructure we have to host. Both fail a hard constraint.

### A.3 Plain tunnel + obscurity (rejected)

The current v0.3 state. Stated user concern is exactly this approach. Out of scope.

### A.4 Cloudflare Zero Trust / Access (rejected)

Free tier exists (~50 users) but: (a) requires the user to set up a Cloudflare account, configure Zero Trust, authenticate via OAuth provider, and (b) bundles WARP client which is a separate app. Fails "no second app, no third party."

### A.5 Cloudflare Tunnel for UDP/WireGuard (rejected — does not exist for free)

Cloudflare Quick Tunnel is HTTP-only. Named tunnels (free, requires CF account) can do TCP/UDP but only inside their Zero Trust private network, which requires every client to run the WARP agent. Not embeddable.
