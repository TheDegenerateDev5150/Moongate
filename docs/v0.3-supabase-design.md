# Moongate v0.3.0 — Supabase-Backed Identity & Access Design

> **Status:** Draft. Local-only. Do not push to GitHub until smoke test passes. Existing v0.2.x users must not be forced to upgrade.

---

## 1. Why this exists

v0.2.x ties identity to *the Pi*: the plugin mints its own `mg_token` JWTs and remembers paired devices in `~/.config/moongate/tokens.json`. That works for a single phone + single Pi, but:

- A new phone can't see the user's printers without re-pairing
- The pairing URL itself is the keys-to-the-kingdom (v0.2.29 incident: leaked pair URL → attacker claims printer)
- There's no central "this person owns this printer" record, so multi-printer / multi-device UX is fragile
- Tunnel URL rotation depends on the phone polling — stale URLs are common after Pi reboots

v0.3.0 keeps the existing transport (Cloudflare Quick Tunnel + token-gated plugin) but moves identity and the printer registry to Supabase. Each app install gets a unique anonymous Supabase user. Printer ownership lives in Postgres with Row-Level Security. The tunnel URL is encrypted at rest and only handed back to the verified owner.

---

## 2. Hard constraints

1. **No GitHub push during development.** All commits stay on the local `v0.3-supabase` branch.
2. **v0.2.x users are not forced to upgrade.** `APK/latest_version.json` continues to point at the v0.2.x stable APK until v0.3.0 is explicitly released.
3. **No IP addresses or sensitive identifiers leak.** Tunnel URLs are encrypted at the field level in Postgres. No IPs returned to clients. No IPs in Edge Function logs.
4. **Cross-user data isolation enforced at the database layer**, not just the application layer. Row-Level Security is non-negotiable.
5. **Solo identity model.** No sharing, no roles, no invitations in v0.3.0. One install = one user = one set of printers.

---

## 3. Threat model

### In scope

| Threat | v0.2.x exposure | v0.3.0 mitigation |
|---|---|---|
| Pairing URL leaks (the v0.2.29 incident) | URL alone could claim a printer | Enrollment token is one-shot, short-TTL, server-validated; `/claim` refuses already-paired printers |
| Tunnel URL leaks | URL + lack of `mg_token` = no control, but URL discloses printer existence | URL is encrypted at rest in Postgres; only handed back to the verified owner over TLS |
| Scanner sweeps `*.trycloudflare.com` | Hits exist but return 401 | Same protection plus auth tied to Supabase identity instead of long-lived per-Pi tokens |
| Cross-tenant data leak (Bob sees Alice's printer) | N/A — each install is isolated by lack of any server | Four walls: app code, Edge Function `auth.uid()` filter, RLS at DB level, Pi-side `user_id` check on every command |
| Stolen phone / lost JWT | 30-day token good until manually revoked | Tokens short-lived (~5 min); refresh tied to Supabase session; remote revocation by deleting Supabase user |
| Enumeration via differential errors | N/A | All Edge Function responses for unknown / unauthorized printer IDs return identical `404` (constant-time) |

### Out of scope (intentional)

- **Moonraker zero-days** on the Pi: token-gating doesn't help if a bug exists pre-auth in Moonraker itself. Mitigation is keeping Moonraker up to date. The full bouncer architecture in v0.4 would address this.
- **Nation-state / targeted attackers:** the threat model is "random opportunistic attackers," not "skilled adversary targeting this user."
- **Malware that escalates and reads Android Keystore:** out of scope. If the OS is compromised, all bets are off.
- **Supabase service compromise:** very low probability and outside our control. Treated as same trust level as Cloudflare.

### Trust boundaries

- **App ↔ Supabase:** TLS, Supabase JWT (refresh token in Keystore)
- **Pi ↔ Supabase:** TLS, Pi device key signs heartbeats
- **App ↔ Pi (via tunnel):** TLS via Cloudflare, short-lived Supabase-issued JWT in `Authorization` header
- **Pi-internal:** Moonraker auth assumes localhost trust; tunnel-facing path enforced by plugin middleware

## 5. Database schema

All tables live in the `public` schema unless noted.

### 5.1 `printers`

```sql
CREATE TABLE printers (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_user_id   uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  name            text NOT NULL,
  pi_public_key   text NOT NULL,                  -- Pi-generated Ed25519 pubkey, used to verify heartbeats
  tunnel_url_enc  bytea,                          -- AES-GCM(tunnel_url); NULL until first heartbeat, see §6
  tunnel_url_nonce bytea,                         -- AES-GCM nonce (96-bit, unique per encrypt); NULL with tunnel_url_enc
  last_seen       timestamptz,
  created_at      timestamptz NOT NULL DEFAULT now(),
  revoked_at      timestamptz,                    -- soft delete; row stays for audit
  CONSTRAINT tunnel_url_both_or_neither
    CHECK ((tunnel_url_enc IS NULL) = (tunnel_url_nonce IS NULL))
);

CREATE INDEX printers_owner_idx ON printers (owner_user_id) WHERE revoked_at IS NULL;
CREATE UNIQUE INDEX printers_pi_pubkey_idx ON printers (pi_public_key) WHERE revoked_at IS NULL;
```

### 5.2 `enrollment_tokens`

One-shot codes minted when a Pi runs `MOONGATE_PAIR`. Used to bind a Pi to an anonymous Supabase user.

```sql
CREATE TABLE enrollment_tokens (
  token_hash      bytea PRIMARY KEY,              -- SHA-256 of the token; raw token never stored
  pi_public_key   text NOT NULL UNIQUE,
  expires_at      timestamptz NOT NULL,
  used_at         timestamptz,
  used_by_user_id uuid REFERENCES auth.users(id)
);

CREATE INDEX enrollment_tokens_expires_idx ON enrollment_tokens (expires_at);
```

### 5.3 Row-Level Security policies

```sql
ALTER TABLE printers ENABLE ROW LEVEL SECURITY;

-- Users see only their own non-revoked printers
CREATE POLICY "select own printers"
  ON printers FOR SELECT
  USING (owner_user_id = auth.uid() AND revoked_at IS NULL);

-- Inserts only via Edge Functions (uses service role), never client direct
-- We explicitly do NOT grant INSERT/UPDATE/DELETE to anon role
REVOKE INSERT, UPDATE, DELETE ON printers FROM anon, authenticated;

ALTER TABLE enrollment_tokens ENABLE ROW LEVEL SECURITY;

-- Anon role has NO access to enrollment_tokens at all
-- All access goes through Edge Functions with service role
REVOKE ALL ON enrollment_tokens FROM anon, authenticated;
```

**Why no client writes:** the app never writes to Postgres directly. All writes go through Edge Functions, which:
1. Verify the Supabase JWT
2. Apply business logic (enrollment token validation, ownership checks)
3. Use the service role to write

This means even if RLS policies have a bug, clients still can't write. The Edge Function is the only write path.

### 5.4 Daily cleanup (pg_cron)

A daily `pg_cron` job runs `moongate_cleanup_inactive()` at 03:15 UTC to prune:
- `printers` rows where `last_seen < now() - interval '6 weeks'` (printer offline for the full window)
- `printers` rows soft-deleted via `revoked_at` more than a week ago
- Expired `enrollment_tokens` rows

The intent is to keep the free-tier database bounded and avoid dead pairings accumulating. Full DDL and Pi-/app-side consequences in §16.

## 6. Field-level encryption strategy

**What's encrypted:** the tunnel URL (only). Everything else in `printers` is either non-sensitive or already a public key.

**Algorithm:** AES-256-GCM. 96-bit random nonce per encrypt. Authentication tag stored as part of the ciphertext output.

**Key management:**

- Master key: a 32-byte random secret, stored as `MOONGATE_TUNNEL_URL_KEY` in Supabase Edge Function environment variables (encrypted at rest by Supabase).
- Edge Functions read the key, encrypt before INSERT/UPDATE on `printers.tunnel_url_enc`, decrypt before returning to the client.
- The key never leaves the Edge Function runtime. Not transmitted to the app. Not transmitted to the Pi.

**Encrypt path** (in Edge Function on heartbeat write):
```ts
const key = await crypto.subtle.importKey("raw", masterKeyBytes, "AES-GCM", false, ["encrypt"]);
const nonce = crypto.getRandomValues(new Uint8Array(12));
const ciphertext = await crypto.subtle.encrypt({ name: "AES-GCM", iv: nonce }, key, encoder.encode(tunnel_url));
// store: tunnel_url_enc = ciphertext, tunnel_url_nonce = nonce
```

**Decrypt path** (in Edge Function on `/access`):
```ts
const key = await crypto.subtle.importKey("raw", masterKeyBytes, "AES-GCM", false, ["decrypt"]);
const plaintext = await crypto.subtle.decrypt({ name: "AES-GCM", iv: nonce }, key, ciphertext);
const tunnel_url = decoder.decode(plaintext);
```

**Rotation plan:** initial release uses one key. If rotation becomes necessary, add a `tunnel_url_key_version` smallint column, dual-write during rotation window, drop old key after all rows re-encrypted. Out of scope for v0.3.0.

**Why this is sufficient:** if Postgres is dumped (SQL injection, accidental backup leak, malicious Supabase staff), tunnel URLs are unreadable without the Edge Function's env var. The threat we're closing is "raw DB access reveals printer locations." Threat we're not closing: "Edge Function runtime is fully compromised" — that's a more involved attack and outside scope.

## 7. Authentication & identity

### 7.1 App identity

- On first launch, app calls `supabase.auth.signInAnonymously()` → receives `{ user, session }` from Supabase
- `session` contains `access_token` (JWT, ~1h expiry) and `refresh_token`
- Both are stored in **Android Keystore** (`EncryptedSharedPreferences` backed by AES key in the StrongBox / TEE)
- Background refresh via `supabase.auth.onAuthStateChange` ensures the access_token never expires mid-use
- The anonymous user's UUID is the `owner_user_id` of every printer they pair

### 7.2 Pi identity

- On plugin first start, generate an Ed25519 keypair (`~/.config/moongate/device_ed25519`)
- Private key never leaves the Pi. Public key is included in the pairing QR and stored in `printers.pi_public_key`
- Pi signs every heartbeat payload with its private key; Edge Function verifies against the stored public key

### 7.3 Access token (Pi-facing)

- Issued by `/printer/{id}/access` Edge Function
- Format: JWT signed with the Supabase JWT secret (same key Supabase uses for its own tokens — Pi can use the same JWKS to verify)
- Claims:
  - `sub` = user UUID
  - `printer_id` = printer UUID
  - `iat`, `exp` (5 minutes from now)
  - `jti` = random UUID
  - `aud` = `"moongate-printer"` (distinguishes from regular Supabase user JWTs)
- Plugin verifies signature against JWKS, checks `aud`, checks `exp`, checks `printer_id` matches its own ID, checks `sub` matches its stored `owner_user_id`

### 7.4 Pi-side owner storage

After successful pairing, plugin stores:
```json
// ~/.config/moongate/owner.json
{
  "owner_user_id": "a3f9...",
  "printer_id": "b2e8...",
  "paired_at": "2026-05-26T19:00:00Z"
}
```
Plugin rejects any access_token whose `sub` doesn't match this `owner_user_id`. This is the **fourth wall** of cross-tenant isolation — even if Supabase somehow issued a wrong token, the Pi refuses it.

## 8. Edge Function contracts

All functions live at `supabase/functions/` and are deployed via `supabase functions deploy`. All use the Supabase service role key (env var) for DB writes after their own auth check.

### 8.1 `POST /printer/claim`

**Purpose:** First-time pairing. Phone has scanned QR, has enrollment token, wants to claim printer.

**Auth:** Caller must present a valid Supabase user JWT (anonymous or otherwise).

**Request body:**
```json
{
  "enrollment_token": "raw_token_string",
  "pi_public_key": "base64-ed25519-pubkey",
  "name": "Bob's Voron"
}
```

**Behavior:**
1. Verify caller's JWT, extract `user_id`
2. Hash enrollment_token (SHA-256), look up in `enrollment_tokens`
3. Constant-time checks:
   - Token exists?
   - Not expired (`now() < expires_at`)?
   - Not used (`used_at IS NULL`)?
   - `pi_public_key` matches the row?
4. If any check fails → 404 (do NOT differentiate "expired" vs "used" vs "not found")
5. Check no existing non-revoked `printers` row with this `pi_public_key` → if exists, 409 (already paired; user must use `RESET_OWNER` on the Pi)
6. Atomic transaction:
   - INSERT into `printers` with `owner_user_id = user_id`, empty `tunnel_url_enc` (filled by next heartbeat)
   - UPDATE `enrollment_tokens` SET `used_at = now()`, `used_by_user_id = user_id`
7. Return `{ printer_id }`

**Rate limit:** 5 attempts per IP per minute (Supabase native rate limiting on functions).

**Error codes:** 401 (no/bad JWT), 404 (any enrollment validation failure), 409 (already paired), 429 (rate limited). Never 400 with detailed reason.

### 8.2 `POST /printer/{id}/access`

**Purpose:** Phone wants to call the Pi. Needs fresh tunnel URL + short-lived token.

**Auth:** Supabase user JWT.

**Request body:** none.

**Behavior:**
1. Verify JWT, extract `user_id`
2. SELECT printer WHERE `id = {id}` (RLS naturally filters to `owner_user_id = auth.uid()`)
3. If no row → 404 (covers both "doesn't exist" and "not yours" — constant-time)
4. If `tunnel_url_enc IS NULL` → 503 with `{retry_after: 5}` (Pi has not yet sent a heartbeat; common in the seconds immediately after `/claim`)
5. Decrypt `tunnel_url_enc` using master key
6. Mint access_token JWT (claims per §7.3), sign with Supabase JWT secret
7. Return `{ tunnel_url, access_token, expires_in: 300 }`

**Note:** `last_seen` on `printers` is NOT updated here. That field tracks Pi liveness via heartbeats, not phone activity, to avoid information leak about user habits.

### 8.3 `POST /printer/{id}/heartbeat`

**Purpose:** Pi reports its current tunnel URL on reboot or rotation.

**Auth:** NOT Supabase JWT. Instead, request is signed with the Pi's Ed25519 private key.

**Request body:**
```json
{
  "tunnel_url": "https://abc-def.trycloudflare.com",
  "timestamp": 1748284800,
  "signature": "base64-ed25519-sig"
}
```

**Behavior:**
1. SELECT printer WHERE `id = {id}` (uses service role, bypasses RLS — but we still don't return anything to the Pi)
2. If no row → 404
3. Verify signature: `ed25519_verify(pi_public_key, signature, json({tunnel_url, timestamp}))`
4. Check `|now() - timestamp| < 60s` (prevents replay)
5. Encrypt `tunnel_url` with master key (§6)
6. UPDATE printer SET `tunnel_url_enc = ?, tunnel_url_nonce = ?, last_seen = now()`
7. Return 204 No Content

**Why Pi doesn't use Supabase Auth:** the Pi doesn't have a user account. Its identity is its device keypair. Treating it as a "user" in Supabase would complicate the auth model unnecessarily.

### 8.4 `POST /printer/enroll` (Pi-side, optional)

If we want enrollment_tokens generated server-side (rather than locally on the Pi as in v0.2.x):

- Pi calls this on `MOONGATE_PAIR` with its public key, signed
- Server generates a random enrollment_token, hashes it, stores hash + pi_public_key + expires_at, returns raw token to Pi
- Pi embeds raw token in QR

Alternative: Pi generates locally (faster, no network dependency during pairing), POSTs hash to Supabase. Both work. **Decision: Pi generates locally** — preserves "pairing works offline" property and lets us reuse the existing `GATE-XXXX-XXXX` format. Pi POSTs the hash to `/printer/enroll-prepare` so the Edge Function can recognize it on `/claim`.

## 9. Plugin (Pi) changes

File: `klipper-plugin/moongate_standalone.py`

### 9.1 Replace `mg_token` validation with Supabase JWT verification

Current code validates HS256 JWTs against a locally-stored secret. v0.3.0:

- On first start, fetch Supabase JWKS from `https://{project}.supabase.co/auth/v1/keys` (cache in memory + disk, refresh hourly)
- For every tunnel-facing request:
  1. Extract token from `Authorization: Bearer <token>` header (preferred) or `?mg_token=` query param (fallback if header not accessible via Moonraker WebRequest API — to be confirmed during Phase 3)
  2. Verify signature against JWKS
  3. Check `aud == "moongate-printer"`
  4. Check `exp > now()`
  5. Check `printer_id == our_printer_id`
  6. Check `sub == our_owner_user_id`
  7. Any failure → 401

### 9.2 Request authentication scope — DECISION

**v0.3.0 ships with Option D: plugin-endpoint-only enforcement.** The `/server/moongate/*` endpoints verify EdDSA JWTs; the underlying Moonraker / Mainsail / Fluidd HTTP surface remains reachable through the tunnel as in v0.2.x.

**Why:**
- The app's control + status + webcam flows ALL go through `/moongate/*` endpoints. Those are now token-gated.
- The Mainsail-via-tunnel exposure surface is **unchanged from v0.2.x** — not a regression; just not the v0.3.0 improvement either.
- The v0.2.29 incident threat ("leaked URL → attacker controls printer") is fully closed: control commands now require an EdDSA-signed access token that only the legitimate owner's app can obtain through Supabase.

**What remains out of scope for v0.3.0:**
- An attacker who somehow learns the tunnel URL can still browse Mainsail and probe native Moonraker endpoints (read printer state, fingerprint Moonraker version). No control commands work without a token.

**Deferred to v0.3.1 / v0.4 — full global middleware.** Two viable approaches when we get there:

**Option A — Moonraker middleware registration:**
Register the plugin as a Tornado request preprocessor that intercepts all HTTP/WS traffic. Needs investigation of Moonraker's component API.

**Option B — Local proxy (preferred):**
- Bind Moonraker to `127.0.0.1:7125` only
- Run a small plugin-owned aiohttp proxy on the tunnel-facing port
- Proxy verifies the EdDSA JWT, forwards to local Moonraker (including WebSocket upgrades)
- `cloudflared` points to the proxy port, not Moonraker

The proxy is more code (~150 lines plus WebSocket forwarding), but it's deterministic across Moonraker versions and gives one auditable choke point. Worth doing once v0.3.0 is stable.

### 9.3 Tunnel URL reporting

- New systemd-level hook or polling: detect cloudflared URL changes (read from `cloudflared` log or status endpoint)
- On detect, POST to `/printer/{id}/heartbeat` with signed payload
- On plugin startup, send one heartbeat immediately

### 9.4 `MOONGATE_RESET_OWNER` macro

Klipper macro that, when triggered, calls a plugin endpoint reachable **only from `127.0.0.1` / LAN** (rejected if request came in via the tunnel):

1. Delete `~/.config/moongate/owner.json`
2. POST to Supabase `/printer/{id}/revoke` (Pi-signed) to mark printer revoked
3. Regenerate enrollment token for next pairing

**Why LAN-only:** prevents an attacker with a valid access_token from triggering a reset to take over the printer.

### 9.5 Backward-compat / coexistence

The plugin will need to detect whether it's a v0.2.x or v0.3.0 install:
- If `~/.config/moongate/owner.json` exists → v0.3.0 mode (Supabase auth)
- If only legacy `tokens.json` exists → v0.2.x mode (mg_token)
- On `install.sh --version 0.3`, migrate or fresh-install

For v0.3.0, the simplest path is **fresh install required**. Old `tokens.json` is left alone but ignored; user must re-pair via the app's new pairing flow. We will not auto-migrate v0.2.x tokens.

## 10. App (Flutter) changes

### 10.1 Dependencies

Add to `mobile/pubspec.yaml`:
- `supabase_flutter: ^2.x` — official Supabase client
- `flutter_secure_storage: ^9.x` — encrypted Keystore wrapper (replaces shared_preferences for sensitive data)

### 10.2 Initialization (`main.dart`)

```dart
await Supabase.initialize(
  url: 'https://{project}.supabase.co',
  anonKey: 'eyJ...',  // Supabase anon key, baked into APK (not sensitive)
);

// On first launch, ensure we have a session
final session = Supabase.instance.client.auth.currentSession;
if (session == null) {
  await Supabase.instance.client.auth.signInAnonymously();
}
```

The `anonKey` is safe to bake into the APK; it grants only anon-role permissions, which we lock down via RLS.

### 10.3 Session persistence

`supabase_flutter` handles refresh tokens automatically using `flutter_secure_storage`. We just point it at the right storage backend in `Supabase.initialize()`.

### 10.4 Access flow

Replace the existing "try local IP, fall back to tunnel" logic with:

```dart
Future<PrinterAccess> getAccess(String printerId) async {
  final response = await Supabase.instance.client.functions.invoke(
    'printer-access',
    body: {'printer_id': printerId},
  );
  return PrinterAccess.fromJson(response.data);
}

Future<http.Response> sendCommand(String printerId, String path, ...) async {
  final access = await getAccess(printerId);
  return http.post(
    Uri.parse('${access.tunnelUrl}$path'),
    headers: {'Authorization': 'Bearer ${access.accessToken}'},
    body: ...,
  );
}
```

**Caching:** access tokens are valid 5 min. Cache per-printer in memory, refresh when within 30s of expiry. Don't persist to disk (short-lived; recompute on app launch).

### 10.5 Listing user's printers

Direct Supabase query (RLS enforces isolation):

```dart
final printers = await Supabase.instance.client
  .from('printers')
  .select('id, name, last_seen')
  .order('created_at');
```

`tunnel_url_enc` is NOT selected — clients never need it. Only `/printer/{id}/access` returns the decrypted URL, and only momentarily.

### 10.6 Pairing flow (app side)

1. User taps "Add Printer"
2. Camera opens, scans QR
3. QR payload: `{pi_id, pi_public_key, enrollment_token}` — three fields; phone never calls the Pi directly during pairing, so tunnel URL is not needed in the QR
4. App calls Edge Function `/printer/claim` with `{enrollment_token, pi_public_key, name}`
5. On 200 → printer added to user's list; navigate to dashboard
6. App makes a `/printer/{id}/access` call to warm the connection. If it returns 503 (no heartbeat yet), retry every 5s up to 30s
7. On 404 → "Pairing code expired or already used. Run MOONGATE_PAIR on the printer again."
8. On 409 → "This printer is already paired with another account. Run MOONGATE_RESET_OWNER on the Pi to reset it."

### 10.7 Remove old code paths

- Delete local-IP discovery / fallback logic
- Delete `mg_token` storage and retrieval
- Delete tunnel URL polling from status responses
- Single transport: Supabase → tunnel

This is a large diff. Plan to feature-flag the new path behind a `kV03Mode` constant initially, so the old code stays compilable until the new path is verified.

## 12. Migration & coexistence with v0.2.x

### 12.1 Goals (per user constraint)

- v0.2.x users on existing installs continue to work indefinitely with their current pairings
- No mandatory update prompt
- v0.3.0 is a separate, opt-in install path
- Once user is happy with v0.3.0 locally, decide release strategy in Phase 7

### 12.2 What stays on the v0.2.x branch / `master`

- `APK/Moongate-latest.apk` → keeps pointing at the latest v0.2.x build
- `APK/latest_version.json` → `"version": "0.2.29"` (or higher v0.2.x patch versions if needed)
- The v0.2.x plugin (`klipper-plugin/moongate_standalone.py` on `master`) untouched
- `master` branch remains the public-facing release

### 12.3 What lives on the `v0.3-supabase` branch

- All v0.3.0 work: schema, Edge Functions, plugin rewrite, app rewrite, this doc
- New `klipper-plugin/v0.3/moongate_standalone.py` (rename or directory separation)
- New `mobile/lib/...` paths with v0.3 services
- Local APK builds only; never committed to `APK/` on this branch until release

### 12.4 In-app upgrade warning (when we eventually release)

When v0.3.0 ships, the app's "Check for updates" feature must explicitly warn:

> **Version 0.3.0 requires re-pairing all printers.**
> Your existing pairings will not carry over. Continue with v0.2.x if you'd rather wait.
> [Stay on v0.2.x] [Upgrade]

Implementation: `latest_version.json` gains an optional `requires_repair: true` flag; current app reads it and shows the warning.

### 12.5 v0.2.x and v0.3.x on the same Pi

Mutually exclusive — the plugin is one file. A Pi running v0.2.x can be upgraded to v0.3.0 by running the new `install.sh`, which:
1. Stops Moonraker
2. Replaces the plugin
3. Wipes `~/.config/moongate/` (old `tokens.json` is obsolete)
4. Restarts Moonraker
5. Tells user to scan QR with v0.3.0 app

A Pi on v0.3.0 cannot serve v0.2.x apps. This is the "hard cutover" trade-off; we accepted it in scope decisions.

## 14. Open questions to settle during implementation

1. **Moonraker middleware feasibility (Phase 3):** Can a plugin globally intercept all HTTP/WS requests, or do we need the proxy approach? Verify before writing the plugin.
2. **`Authorization` header vs query param:** Does Moonraker's request API expose headers to components? v0.2.x security.md says no. If still true, JWTs go in query params (`?token=`). Less clean but works.
3. **JWKS endpoint:** Does Supabase expose a stable JWKS URL for verifying signed JWTs? If not, we may need to use HS256 with a shared secret between Edge Functions and the Pi — less ideal but workable.
4. **Anonymous user lifetime:** Supabase has a 24h default for anonymous sessions; we need to confirm refresh tokens extend this indefinitely as expected.
5. **Pi keypair persistence across re-installs:** Should `install.sh` preserve `device_ed25519` or always regenerate? Probably preserve — otherwise every reinstall requires re-pairing.
6. **Rate limiting on `/access`:** A compromised app could spam this endpoint. Decide acceptable RPS per user.

These will be resolved as their phases are reached. Document the decisions in this file as they're made.

## 15. Glossary

- **Anonymous Supabase user:** A real Supabase auth user with no email, identified only by UUID. Created via `signInAnonymously()`.
- **Enrollment token:** One-shot, short-TTL, single-use code shown in QR on the Pi. Used to bind a Pi to the scanning user.
- **Access token:** Short-lived (5 min) JWT minted by Edge Function, presented to Pi on every command call.
- **Heartbeat:** Pi-signed POST to Supabase reporting current tunnel URL. Sent on plugin start and on `cloudflared` URL change.
- **RLS:** Row-Level Security — Postgres feature that filters rows by policies at query time. Defense-in-depth for multi-tenancy.
- **JWKS:** JSON Web Key Set — public keys for verifying signed JWTs.
- **pg_cron:** Postgres extension for scheduling SQL jobs (cron-like) inside the database. Supabase free tier supports it.

## 16. Data retention & cleanup

Free-tier Supabase (~500 MB Postgres, ample for this project) still benefits from active hygiene to keep the table small, response times fast, and stale data out of backups. A daily `pg_cron` job handles it.

### 16.1 Retention policies

| Data | Trigger for deletion | Why |
|---|---|---|
| `printers` row, `revoked_at IS NULL` | `last_seen < now() - interval '6 weeks'` (or `last_seen IS NULL AND created_at < now() - interval '6 weeks'`) | Pi has missed every heartbeat for 6 full weeks — pairing is effectively dead |
| `printers` row, soft-deleted | `revoked_at < now() - interval '1 week'` | One-week audit window after a `RESET_OWNER`, then prune |
| `enrollment_tokens` row | `expires_at < now() - interval '1 day'` | Tokens have minutes-long TTL; a day of grace then they're dead weight |
| `auth.users` (anonymous, no remaining printers) | **Deferred to future work** — see §16.6 | Cleanest deletion requires Edge-Function-mediated `auth.admin.deleteUser()`; not worth the complexity at this scale |

A single Pi heartbeat anywhere in the 6-week window resets `last_seen` and saves the printer. The 6-week threshold only catches truly abandoned setups.

### 16.2 Function + schedule

```sql
-- One-time: enable extension (Supabase dashboard → Database → Extensions, or via SQL)
CREATE EXTENSION IF NOT EXISTS pg_cron;

CREATE OR REPLACE FUNCTION moongate_cleanup_inactive()
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  deleted_stale   int;
  deleted_revoked int;
  deleted_tokens  int;
BEGIN
  -- 1. Stale printers (no heartbeat in 6 weeks)
  DELETE FROM printers
  WHERE revoked_at IS NULL
    AND (
      (last_seen IS NULL AND created_at < now() - interval '6 weeks')
      OR last_seen < now() - interval '6 weeks'
    );
  GET DIAGNOSTICS deleted_stale = ROW_COUNT;

  -- 2. Old soft-deleted printers
  DELETE FROM printers
  WHERE revoked_at < now() - interval '1 week';
  GET DIAGNOSTICS deleted_revoked = ROW_COUNT;

  -- 3. Expired enrollment tokens (1-day grace)
  DELETE FROM enrollment_tokens
  WHERE expires_at < now() - interval '1 day';
  GET DIAGNOSTICS deleted_tokens = ROW_COUNT;

  RAISE NOTICE 'moongate_cleanup: % stale, % revoked, % tokens',
    deleted_stale, deleted_revoked, deleted_tokens;
END;
$$;

-- Daily at 03:15 UTC (low-traffic window for most timezones)
SELECT cron.schedule(
  'moongate-cleanup-inactive',
  '15 3 * * *',
  $$ SELECT moongate_cleanup_inactive(); $$
);
```

### 16.3 Pi-side consequences

If cleanup deletes a `printers` row while the Pi still has `~/.config/moongate/owner.json` on disk, the next heartbeat returns 404 from `/printer/{id}/heartbeat`. The plugin must handle this gracefully:

- 404 on heartbeat → assume printer record is gone server-side
- Wipe local `owner.json` and tunnel-URL cache
- Display "Printer unpaired — run `MOONGATE_PAIR` to re-pair" on the plugin status endpoint
- All subsequent control requests already fail at JWT verification (no stored owner)

A long-offline Pi (vacation, broken cooler, indefinite shelf storage) thus gracefully degrades to "needs re-pairing" rather than getting stuck in a half-paired limbo.

### 16.4 App-side consequences

If a user's printer is cleaned up while their app is installed:
- Next `/printer/{id}/access` call returns 404
- App removes the printer from local cache and shows empty state: "No printers paired. Tap + to add one."

The user themselves is **not** deleted (per §16.1), so they retain their anonymous identity. If they re-pair the same physical Pi, it's a fresh row but the same `owner_user_id`.

### 16.5 Observability

`RAISE NOTICE` output appears in Supabase dashboard → Database → Logs → Postgres logs. Casual daily check is enough — sudden spikes would indicate something pathological.

Optional future addition: a `cleanup_log` table that records counts per run with a 90-day TTL. Out of scope for v0.3.0.

### 16.6 Future: orphan anonymous user cleanup

Deferred to a later version. Implementing it properly requires:
1. Enable `pg_net` extension
2. Create an Edge Function endpoint that calls `supabase.auth.admin.deleteUser(id)` (cleans up sessions, refresh tokens, identities — direct `DELETE FROM auth.users` would leave orphans)
3. Have `moongate_cleanup_inactive()` issue `net.http_post` to that endpoint for each candidate user

For free-tier scale, orphan `auth.users` rows are ~few hundred bytes each — even 100K orphans is ~50 MB. Tolerable. Revisit if usage actually justifies it.
